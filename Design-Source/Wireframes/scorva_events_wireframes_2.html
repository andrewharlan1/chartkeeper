Scorva — Events Wireframes

Wireframe · v0.1 · Apr 30 2026
Supplements the player view wireframes with the events / setlists feature. From Jason Buchanan interview: charts also need to be organized by performance, gig, or rehearsal. An event is an ensemble-scoped object that references charts (charts are not removed from the chart list when added to an event — they appear in both). Players gain a new pivot in My Parts: by chart / by ensemble / by event. The player view itself gets an event-context bar when accessed via a setlist.

Main page is the chart-card grid (matches Claude Design image 2) with Roster & Instruments below. Members & Events live in a slide-in panel that opens from a button in the topbar — keeps the main view focused on the work, surfaces the context on demand.

Design notes

  • Main page matches Claude Design image 2: chart-card grid, header stats, Roster & Instruments table below. The card thumbnails carry "X CHANGED" / "DRAFT" / "ARCHIVED" badges in the top-right corner.
  • The "Members & events" trigger lives in the topbar with a small accent dot when an event is imminent. Tap to open the slide-in panel (next screen, E1b).
  • assume Trigger label is "Members & events" — could also be a single icon if topbar gets crowded.
  • Stats in the header (3 / 5 / 2) repeat what's elsewhere on the page but give the director a glance summary. Identical to image 2.
  • Roster row layout matches image 2: instrument icon, instrument name (with "score lane" pill or "you" pill), assigned member with avatar + tenure, action buttons. The score lane gets the gradient background to set it apart.
  • defer Drag-to-reorder roster rows. The "drag to reorder" hint can be added in a polish pass.

Design notes

  • Slide-in is 360px wide. The page behind dims slightly (rgba 0.18 ink) so focus is on the panel without losing context.

  • The trigger button in the topbar gets a subtle accent state when the panel is open (shown here in ink-on-ink), to mirror selected-state.

E6 · Player view with event context

When the player opens a chart through an event (rather than directly), a thin event-context bar sits between the topbar and the toolbar. It shows the event name, position in the setlist (3 of 8), and a "next chart" affordance. Tapping "next" advances the player to chart 4 in the setlist. Without this context, the player view is identical to the standalone version.

Design notes

  • The event-context bar is only present when the chart was opened via an event. Direct chart access keeps the player view as designed (no event bar).
  • "Next chart" advances to setlist position N+1. "Long-press the foot pedal" is a future affordance for hands-free chart switching during performance — relevant for solo-with-pedal contexts.
  • assume The breadcrumb leads with "April 30 Gig" instead of "Lunar Noon" because the user's mental context is the event.
  • assume "Time remaining" is the sum of remaining setlist durations. Useful before downbeat for pacing.
  • defer Auto-advance after a chart's last page (with confirmation). Risky during performance — hand input only for v1.
  • defer "Mark this chart as played" affordance for a director's tracking view post-performance.
